# OpsPilot AI 项目说明（面试版）

## 1. 项目一句话介绍

`OpsPilot AI` 是一个面向团队 OnCall 场景的智能运维助手项目，核心目标是解决内部文档分散、问题排查依赖经验、告警处理效率低的问题。

它不是一个单纯的聊天机器人，而是把以下三类能力整合到一个系统里：

- 知识库检索
- 对话式交互
- 运维工具调用与告警分析

目标是让常见问题能自助应答，让复杂告警能辅助定位，从而降低团队的人力成本和响应压力。


## 2. 业务背景与解决的问题

在很多 OnCall 团队里，常见问题通常有几个痛点：

- 内部知识分散在文档、SOP、群消息和个人经验里
- 告警来了以后，经常要人工查文档、翻日志、看监控，排查路径不稳定
- 同样的问题会被重复问、重复查，信息复用率低
- 新人缺少上下文，处理效率和准确率都受影响

这个项目的定位，就是把“知识、对话、运维证据”三条链路打通，形成一个能落地的智能 OnCall Assistant。


## 3. 我怎么定义这个项目

如果用面试里更容易理解的方式来讲，我会把它定义成：

> 一个基于 RAG、Agent 编排和运维工具接入的智能 OnCall 系统原型。

它最重要的价值，不是单次模型问答，而是把下面这条链路串起来：

```text
用户问题 / 告警输入
-> 知识检索
-> 工具调用
-> 结果整合
-> 交互式输出
-> 辅助决策
```

所以它更像一个“可工程化演进的运维助手骨架”，而不是一个简单的聊天 Demo。


## 4. 系统能力拆分

我会把这个项目分成四块来讲。

### 4.1 知识层：RAG 知识库系统

知识层负责把内部资料变成“可检索上下文”，主要包括：

- 文档上传
- 文档切片
- 向量化
- Milvus 存储
- 相似检索

这里我重点关注两个参数：

- `chunk` 切分粒度
- `TopK` 检索数量

因为这两个参数直接影响检索质量：

- `chunk` 太小，会丢失上下文
- `chunk` 太大，会把噪声带进来
- `TopK` 太大，会污染上下文，也会拖慢性能

这里我更倾向把它描述成：

- `RAG 索引与检索模块`
- `知识库索引流水线`

而不是独立 Agent。

原因很简单：

- 文档上传、切片、向量化、入库本质上是确定性的工程流水线
- 这类流程更适合做成 service / async task
- 没必要为了“看起来更智能”而强行 agent 化


### 4.2 对话层：ReAct 问答 Agent

普通问答场景，我使用的是 `ReAct` 思路。

这里的特点是：

- 用户发起自然语言问题
- Agent 根据 prompt 判断要不要调用工具
- 如果需要，就调用知识库检索、日志、告警等工具
- 再基于工具结果生成回答

这种模式适合：

- 常见问答
- 内部知识查询
- 简单工具调用
- 单轮或轻量多轮交互

我没有把普通问答做成复杂的多 Agent 工作流，因为这类任务本身并不需要显式的“规划-执行-再规划”，单个 ReAct Agent 已经足够。


### 4.3 AIOps 层：Plan-Execute-Replan 多 Agent 工作流

告警分析这类任务和普通问答不一样，它往往天然是一个多步任务：

- 先判断应该先查什么
- 再执行一步
- 看结果是否支持假设
- 如果不支持，就调整方向
- 最后输出处理建议或分析报告

所以在这一层，我采用的是 `Plan-Execute-Replan` 思路。

当前项目里，这个工作流由三部分组成：

- `Supervisor`
- `Planner`
- `Executor`

其中：

- `Supervisor` 负责总调度
- `Planner` 负责拆解和更新排查步骤
- `Executor` 负责执行当前这一步

这个设计的重点不是“把 Agent 拆得越多越好”，而是：

- 把“决策”和“执行”拆开
- 让复杂任务更可控、更稳定、更好调试

另外一个很重要的点是：

> 这个项目的 AIOps 设计是外层 `Plan-Execute-Replan`，内层仍然嵌套 `ReAct`。

也就是说：

- 外层是 workflow orchestration
- 内层的 `Planner` 和 `Executor` 底层仍然是 `ReactAgent`

这点在面试里很重要，因为它说明这个项目不是“ReAct 和 PER 二选一”，而是分层组合使用。


### 4.4 工具层：把证据查询标准化接入

为了让 Agent 不只是“会回答”，而是能够“拿证据”，我把工具能力作为一个独立层接入。

当前项目里比较核心的工具包括：

- 内部知识库检索
- Prometheus 告警查询
- 腾讯云 MCP 日志查询
- 时间工具

工具层的意义在于：

- 让 Agent 能基于实时数据而不是纯靠模型记忆回答
- 把日志、监控、知识等能力统一暴露给 Agent
- 为后续继续接更多工具保留扩展点

如果用更工程化的话说，这一层承担的是：

- `tool abstraction`
- `evidence acquisition`
- `observability integration`


## 5. 当前系统的完整业务链路

这个项目最完整、最能体现价值的一条链路是告警分析。

可以概括成：

```text
收到告警
-> 结合知识库检索相关 SOP / 文档
-> Planner 规划排查步骤
-> Executor 调用日志 / 监控工具收集实时证据
-> Planner 根据反馈再规划
-> 输出结构化分析报告
```

它体现出来的不是单点模型调用，而是一整条工程化链路：

- `RAG`
- `Agent 编排`
- `Tool Calling`
- `SSE 流式交互`
- `AIOps 场景落地`


## 6. 这套设计为什么合理

### 6.1 为什么普通问答用 ReAct

因为普通问答大多数时候是：

- 理解问题
- 判断要不要调用工具
- 调用 1 到数次工具
- 组织回答

这类任务不一定需要显式规划，用单个 ReAct Agent 更轻、更自然，也更容易控制延迟。


### 6.2 为什么告警分析要拆成 Plan-Execute-Replan

因为告警分析明显属于复杂任务：

- 不是单步完成
- 需要多次获取证据
- 需要根据中间结果调整策略
- 最后往往要输出结构化结论

如果把这种任务完全塞给一个 Agent，常见问题会是：

- 计划不清晰
- 工具调用顺序不稳定
- 中间推理过程不可控
- 一旦走偏，很难纠偏

而把它拆成 `Planner / Executor` 后，系统的可控性会更强。


### 6.3 为什么知识入库不应该强行做成 Agent

这个问题其实很容易被面试官追问，所以我会主动说清楚：

- 文档上传
- chunk 切分
- embedding
- 向量入库

这些过程本质上是确定性的，不需要“自主推理”。

因此我更倾向于把它保留为：

- 异步索引任务
- RAG pipeline
- service flow

而不是独立 Agent。

这也体现了一个设计原则：

> 不是所有模块都应该 agent 化，只有需要决策和推理的环节才值得做成 Agent。


## 7. 我在项目里重点做了什么

如果从工程改造角度讲，我重点做了几类事情。

### 7.1 梳理 Prompt 与 Agent 的边界

我把 Prompt 从业务逻辑里拆出来，做成独立的 builder，减少 prompt 和 service 的耦合，让：

- 普通聊天
- Planner
- Executor
- Supervisor

这些提示词都能独立管理。


### 7.2 收口 Agent 装配逻辑

我把原来分散在不同 service 里的 Agent 构建逻辑统一到了：

- `AgentFactory`
- `AgentToolRegistry`

这样 Chat 和 AIOps 的装配方式更一致，也更方便后续继续扩展 specialist agent。


### 7.3 把上传索引改成异步任务

原来文档上传后会同步做 embedding 和 Milvus 入库，体验上容易卡住。

我把它改成：

- 上传先成功返回
- 后台异步执行索引
- 前端轮询任务状态

这样更符合真实业务场景，也更利于后续扩展任务管理。


### 7.4 增强知识库稳定性

我补了 embedding 的重试和错误收敛逻辑，避免 DashScope 偶发超时导致整个索引链路直接失败。


### 7.5 统一异常出口

我补了统一异常处理，让接口错误不再散落在 controller 里，提升系统的一致性和后续维护性。


## 8. 当前项目的优点

如果面试官问“你觉得这个项目做得好的地方是什么”，我会重点说这几条：

- 它不是孤立的大模型调用，而是完整串起了知识、工具和场景
- 普通问答和复杂告警分析用了不同的 Agent 策略，任务分层比较清楚
- RAG、Tool Calling、Multi-Agent Workflow、SSE 这些关键能力都落地了
- 具备一定工程改造基础，而不是完全一次性 Demo


## 9. 当前项目的边界与不足

面试时我也会主动承认一些边界，这会显得更真实：

- 当前仍然是单体 Spring Boot 应用
- `QueryMetricsTools` 和 `QueryLogsTools` 还存在 mock 和真实实现混杂的问题
- 会话和索引任务状态还主要保存在内存中
- AIOps 中间状态还偏文本化，还可以继续 schema 化
- 前端仍然是轻量静态页面，更适合 PoC 和内部演示

这也说明项目现在的定位更偏：

- 可运行的工程骨架
- 面向后续继续工程化改造的中间版本


## 10. 面试时我会怎么总结它

如果要用一段比较完整、又比较自然的话来描述这个项目，我会这样说：

> 这个项目主要解决的是团队 OnCall 场景里文档分散、排查依赖经验、告警处理效率低的问题。它不是一个单纯聊天机器人，而是把知识库检索、对话式交互和运维工具调用整合到一起的智能 OnCall Assistant。  
>  
> 架构上我把它拆成三层。第一层是基于 RAG 的知识层，负责文档切片、向量化、Milvus 存储和相似检索；第二层是 Agent 编排层，普通问答使用 ReAct，告警分析使用 Supervisor、Planner、Executor 组成的 Plan-Execute-Replan 工作流；第三层是工具层，通过知识库检索、Prometheus 告警查询和腾讯云日志查询，把实时证据接入 Agent。  
>  
> 这个项目最有代表性的地方，是把 RAG、Agent 编排、工具调用和 AIOps 场景串成了一条完整的工程化链路。对我来说，它的重点不是“把所有东西都做成 Agent”，而是根据任务复杂度决定哪里需要 ReAct，哪里需要多 Agent workflow，哪里更适合作为确定性的 pipeline 或 tool。


## 11. 一句话定位（适合最后收口）

如果最后只留一句话，我会这么总结：

> `OpsPilot AI` 是一个把 RAG、Tool Calling 和多 Agent 编排落到 OnCall 场景里的智能运维助手原型，重点体现的是完整工程链路，而不是单点模型能力。
